## 技术选择

数据库采用技术选型确认书中的 MySQL 8.0。后端通过 FastAPI 服务编排 TTS 生成与缓存读取，后续可接入 SQLAlchemy 2.0 管理模型和迁移。

MySQL 不直接保存音频二进制。数据库只保存音频元数据、文件路径、访问 URL、生成状态和供应商信息；真实音频文件保存在本地 `backend/static/previews/`，后续可以平滑迁移到 OSS。

## 目标流程

音色试听接口的目标流程是：

1. 前端请求试听，传入业务音色或供应商音色 ID。
2. 后端定位音色对应的 TTS 供应商配置。
3. 后端计算试听文本的 SHA-256 hash。
4. 查询 `voice_preview_audios` 是否已有同供应商、同文本、同格式的缓存。
5. 命中缓存时直接返回 `audio_url`。
6. 未命中时调用 Edge-TTS 或 Qwen-TTS 生成音频文件。
7. 生成成功后写入 `voice_preview_audios`，再返回音频 URL。

## 表结构

### users

`users` 是用户账号表，存储注册用户信息和 bcrypt 密码哈希。

关键字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `username` | `VARCHAR(50)` | 用户名，唯一。 |
| `password_hash` | `VARCHAR(255)` | bcrypt 哈希后的密码，不存明文。 |
| `email` | `VARCHAR(120)` | 邮箱（可选）。 |
| `phone` | `VARCHAR(20)` | 手机号（可选）。 |
| `avatar` | `TEXT` | 头像 base64 或 URL。 |
| `is_active` | `BOOLEAN` | 是否启用。 |

唯一约束：

```text
username
```

### voices

`voices` 是业务音色表，服务前端音色库展示和筛选，不绑定某一个 TTS 平台。

关键字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `voice_key` | `VARCHAR(100)` | 业务音色唯一键，例如 `xiaoxiao`。 |
| `display_name` | `VARCHAR(50)` | 展示名称，例如 `晓晓`。 |
| `gender` | `VARCHAR(20)` | `female`、`male`、`child`。 |
| `style` | `VARCHAR(50)` | 展示风格，例如 `温柔`。 |
| `category` | `VARCHAR(50)` | 展示分类，例如 `知识类`。 |
| `description` | `VARCHAR(255)` | 音色说明。 |
| `is_recommended` | `BOOLEAN` | 是否推荐。 |
| `is_active` | `BOOLEAN` | 是否启用。 |
| `owner_id` | `INT` | 所属用户 ID（`NULL` = 系统默认，非 NULL = 某用户的专属音色）。 |

### voice_provider_profiles

`voice_provider_profiles` 是 TTS 供应商映射表，把业务音色映射到 Edge-TTS、Qwen-TTS 等平台的真实 voice id。

关键字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `voice_id` | `BIGINT` | 关联 `voices.id`。 |
| `provider` | `VARCHAR(50)` | 供应商，例如 `edge_tts`、`qwen_tts`。 |
| `provider_voice_id` | `VARCHAR(120)` | 供应商内的真实音色 ID，例如 `zh-CN-XiaoxiaoNeural`。 |
| `locale` | `VARCHAR(20)` | 语言区域，例如 `zh-CN`。 |
| `supports_wav` | `BOOLEAN` | 供应商是否直接支持 WAV。 |
| `supports_mp3` | `BOOLEAN` | 供应商是否直接支持 MP3。 |
| `is_default` | `BOOLEAN` | 是否为该业务音色默认供应商。 |

唯一约束：

```text
provider + provider_voice_id
```

这样同一个供应商音色不会被重复写入，同时允许一个业务音色同时拥有 Edge-TTS 和 Qwen-TTS 配置。

### voice_preview_audios

`voice_preview_audios` 是试听音频缓存表，保存某个供应商音色在某段试听文本和某种格式下生成出来的音频记录。

关键字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `voice_provider_profile_id` | `BIGINT` | 关联供应商音色配置。 |
| `sample_text_hash` | `CHAR(64)` | 试听文本 SHA-256，避免长文本参与唯一索引。 |
| `sample_text` | `VARCHAR(1000)` | 真实试听文本，便于排查和复用。 |
| `format` | `VARCHAR(10)` | 音频格式，例如 `mp3`、`wav`。 |
| `audio_path` | `VARCHAR(500)` | 后端文件路径或对象存储 key。 |
| `audio_url` | `VARCHAR(500)` | 前端可访问 URL。 |
| `duration_seconds` | `DECIMAL(8,2)` | 音频时长。 |
| `file_size_bytes` | `BIGINT` | 文件大小。 |
| `status` | `VARCHAR(20)` | `ready`、`generating`、`failed`。 |
| `error_message` | `VARCHAR(1000)` | 生成失败原因。 |

唯一约束：

```text
voice_provider_profile_id + sample_text_hash + format
```

这个约束保证同一个供应商音色、同一段试听文本、同一种格式只生成一次。

## Edge-TTS 与 Qwen-TTS 兼容策略

Edge-TTS 的 `provider_voice_id` 使用当前前端里的 ID，例如：

```text
zh-CN-XiaoxiaoNeural
```

Qwen-TTS 后续接入时，不需要修改 `voices` 表，只需要给同一个 `voice_id` 新增一条供应商配置：

```text
provider = qwen_tts
provider_voice_id = Qwen-TTS 的音色 ID
```

试听缓存也会按 `voice_provider_profile_id` 隔离，因此 Edge-TTS 与 Qwen-TTS 可以为同一个业务音色分别生成试听片段，不会互相覆盖。

## 当前个人素材库与个人音色库边界

截至当前版本，数据库中已经存在“个人音色库”的基础结构，但还没有真正的“个人素材库”结构。

### 当前个人音色库

当前个人音色库由两层表表达：

1. `voices`
   - 作为前端音色库展示的业务音色表。
   - `owner_id IS NULL` 表示系统默认音色。
   - `owner_id = users.id` 表示某个用户的个人专属音色。

2. `voice_provider_profiles`
   - 作为业务音色到真实 TTS provider 的映射。
   - 当前主要 provider 是 `edge_tts`。
   - 同一个 `voices.id` 可以挂多个 provider，例如 `edge_tts`、`qwen3_tts`。

也就是说，前端“个人音色库”不应该直接关心 Edge-TTS 或 Qwen3-TTS 的内部模型路径，而是只读取 `voices`。后端在合成时再根据 `voice_provider_profiles` 找到实际 provider 和模型配置。

### 当前个人素材库

当前数据库没有独立的个人素材库表。已有的音频文件主要分为两类：

- `voice_preview_audios`：试听缓存，不是用户上传素材。
- `backend/media/` + `backend/data/works.json`：作品音频和作品元数据，当前作品元数据还没有完全进入 MySQL。

后续接入 Qwen3-TTS 音色克隆时，用户上传的 mp3、wav、m4a 或 zip 数据集不应写入 `voice_preview_audios`，而应进入独立的个人素材库表。数据库只保存元数据、路径、校验状态和归属关系，真实文件仍保存到本地文件系统或对象存储。

## Qwen3-TTS(1.7B) 接入后的推荐数据模型

Qwen3-TTS 音色克隆链路建议拆成四类实体：

1. **素材资产**：用户上传的 mp3/wav/m4a/zip。
2. **训练任务**：一次 Qwen3-TTS 微调或模型导入任务。
3. **模型产物**：微调完成后的模型、LoRA adapter、checkpoint 或导入模型目录。
4. **业务音色**：创作工作台最终可选择的音色，仍然落在 `voices` + `voice_provider_profiles`。

这样做的好处是：素材库负责“原始数据”，训练任务负责“过程”，模型产物负责“可复用模型”，音色库负责“产品展示和合成入口”。四者分离后，后续删除 Edge-TTS 默认音色、切换 Qwen3-TTS、导入已有模型都不会破坏前端选音色流程。

### user_material_assets

`user_material_assets` 是个人素材库主表，记录用户上传或导入的原始素材。素材可以是单个音频，也可以是 zip 压缩包。

关键字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `BIGINT` | 主键。 |
| `user_id` | `BIGINT` | 所属用户，关联 `users.id`。 |
| `asset_type` | `VARCHAR(30)` | 素材类型：`audio`、`dataset_zip`、`transcript`、`other`。 |
| `purpose` | `VARCHAR(50)` | 用途：`qwen3_tts_finetune`、`voice_reference`、`work_material`。 |
| `original_filename` | `VARCHAR(255)` | 用户上传时的原始文件名。 |
| `mime_type` | `VARCHAR(100)` | MIME 类型，例如 `audio/mpeg`、`application/zip`。 |
| `storage_path` | `VARCHAR(500)` | 后端本地路径或对象存储 key。 |
| `storage_url` | `VARCHAR(500)` | 可选访问 URL。 |
| `file_size_bytes` | `BIGINT` | 文件大小。 |
| `sha256` | `CHAR(64)` | 文件哈希，用于去重和审计。 |
| `duration_seconds` | `DECIMAL(10,2)` | 音频总时长，zip 可记录解压后总时长。 |
| `sample_rate` | `INT` | 音频采样率，可为空。 |
| `channels` | `INT` | 声道数，可为空。 |
| `status` | `VARCHAR(30)` | `uploaded`、`validating`、`ready`、`failed`、`deleted`。 |
| `validation_message` | `VARCHAR(1000)` | 校验结果说明。 |
| `consent_confirmed` | `BOOLEAN` | 是否确认拥有声音使用授权。 |
| `consent_version` | `VARCHAR(50)` | 授权声明版本。 |
| `metadata_json` | `JSON` | 扩展元数据，例如 zip 解压摘要、语言、说话人信息。 |
| `created_at` | `DATETIME` | 创建时间。 |
| `updated_at` | `DATETIME` | 更新时间。 |

建议索引：

```text
user_id + status
user_id + purpose
sha256
```

### user_material_items

`user_material_items` 是素材明细表，主要用于 zip 数据集解压后的单条音频样本。若用户只上传单个 mp3，也可以生成一条 item，方便统一训练输入。

关键字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `BIGINT` | 主键。 |
| `asset_id` | `BIGINT` | 关联 `user_material_assets.id`。 |
| `user_id` | `BIGINT` | 冗余用户 ID，便于鉴权查询。 |
| `item_index` | `INT` | 样本序号。 |
| `audio_path` | `VARCHAR(500)` | 解压或转码后的单条音频路径。 |
| `transcript` | `TEXT` | 对应文本，可为空。 |
| `duration_seconds` | `DECIMAL(10,2)` | 单条音频时长。 |
| `sample_rate` | `INT` | 采样率。 |
| `channels` | `INT` | 声道数。 |
| `quality_score` | `DECIMAL(5,2)` | 质量评分，可由后续音频检测任务写入。 |
| `status` | `VARCHAR(30)` | `ready`、`invalid`、`ignored`。 |
| `error_message` | `VARCHAR(1000)` | 无效原因。 |

zip 数据集建议约定结构：

```text
dataset.zip
├── wavs/
│   ├── 0001.wav
│   └── 0002.wav
└── metadata.csv
```

`metadata.csv` 可采用：

```text
filename|text
0001.wav|这里是第一条训练文本
0002.wav|这里是第二条训练文本
```

若用户只上传 mp3 且没有 transcript，则可以作为 zero-shot reference 或弱训练素材；若要做严肃微调，建议补充 transcript。


### voice_provider_profiles 扩展

为了兼容 Edge-TTS 和 Qwen3-TTS，不建议让创作工作台直接选择 `voice_model_artifacts`。更推荐继续使用 `voices` 作为统一业务音色入口，并扩展 `voice_provider_profiles`。

建议新增字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `provider_kind` | `VARCHAR(30)` | `remote_voice`、`local_model`、`finetuned_model`。 |
| `model_artifact_id` | `BIGINT` | 关联 `voice_model_artifacts.id`，Edge-TTS 可为空。 |
| `runtime_config_json` | `JSON` | 推理参数，例如推理服务地址、显存配置、默认采样参数。 |

兼容关系：

| 场景 | `voices` | `voice_provider_profiles` |
| --- | --- | --- |
| Edge-TTS 系统音色 | `owner_id = NULL` | `provider = edge_tts`，`provider_voice_id = zh-CN-XiaoxiaoNeural` |
| Qwen3-TTS 个人微调音色 | `owner_id = 当前用户` | `provider = qwen3_tts`，`model_artifact_id = voice_model_artifacts.id` |
| Qwen3-TTS 导入模型 | `owner_id = 当前用户` | `provider = qwen3_tts`，`provider_kind = local_model`，`model_artifact_id` 指向导入产物 |

若短期不想改 `voice_provider_profiles` 表结构，也可以把 `model_artifact_id`、`provider_kind` 临时放入 `provider_voice_id` 和 JSON 配置中，但长期不推荐，因为会让 provider 解析逻辑变得脆弱。

## 创作工作台统一选音色策略

创作工作台的“选择音色”只应依赖 `/api/voices` 返回的业务音色列表，不应该区分这是 Edge-TTS 音色还是 Qwen3-TTS 模型音色。

推荐返回结构：

```json
{
  "id": 101,
  "voiceKey": "user-12-qwen3-teacher-v1",
  "displayName": "我的讲师音色",
  "gender": "male",
  "style": "专业",
  "category": "知识类",
  "ownerId": 12,
  "providers": [
    {
      "provider": "qwen3_tts",
      "providerKind": "finetuned_model",
      "modelArtifactId": 88,
      "isDefault": true,
      "isActive": true
    }
  ]
}
```

合成时仍调用统一接口：

```text
POST /api/tts/generate
```

后端处理流程：

1. 根据 `voiceId` 找到 `voices`。
2. 校验当前用户是否可见该音色。
3. 找到该音色的默认 `voice_provider_profiles`。
4. 如果 `provider = edge_tts`，调用 Edge-TTS。
5. 如果 `provider = qwen3_tts`，读取 `voice_model_artifacts` 的模型路径，调用 Qwen3-TTS 推理服务。
6. 生成音频后写入作品记录。

这样前端无需为 Edge-TTS 和 Qwen3-TTS 写两套选择逻辑。

## Qwen3-TTS 音色克隆推荐流程

### 1. 上传素材

用户在“个人素材库”上传 mp3/wav/m4a 或 zip：

```text
POST /api/materials
```

后端写入：

- `user_material_assets`
- 如为 zip，解压并写入 `user_material_items`

### 2. 创建训练或导入任务

训练任务：

```text
POST /api/voice-training/jobs
job_type = finetune
provider = qwen3_tts
base_model = qwen3-tts-1.7b
material_asset_id = xxx
```

导入任务：

```text
POST /api/voice-training/jobs
job_type = import_model
provider = qwen3_tts
model_path = /models/user-12/my-voice
```

### 3. 生成模型产物

任务成功后写入：

- `voice_model_artifacts`
- `voices`
- `voice_provider_profiles`

其中 `voices` 用于前端展示，`voice_model_artifacts` 用于后端推理定位模型。

### 4. 创作工作台使用

用户在创作工作台选择该音色，输入待合成文本。后端根据 provider 调用 Qwen3-TTS 推理服务，无需再次训练。

## Qwen3-TTS 与 Edge-TTS 去留策略

现有 Edge-TTS 音色后续如果要删除，不建议物理删除历史表数据。推荐策略：

1. 对系统默认 Edge-TTS 音色设置：

```text
voices.is_active = false
voice_provider_profiles.is_active = false
```

2. 保留历史作品和试听缓存：

```text
voice_preview_audios 不删除
works 不删除
```

3. 新建 Qwen3-TTS 音色：

```text
voices.owner_id = 当前用户或 NULL
voice_provider_profiles.provider = qwen3_tts
voice_provider_profiles.model_artifact_id = voice_model_artifacts.id
```

4. 创作工作台默认只展示 `is_active = true` 的音色。

这样可以避免旧作品音频、缓存记录和用户收藏突然失效。

## 音频格式策略

试听场景推荐优先使用 MP3，因为体积小、加载快，适合前端播放。WAV 适合作品导出或质量检查。

如果业务要求试听缓存使用 WAV，后端流程应为：

1. Edge-TTS 先生成 MP3。
2. 使用 pydub + FFmpeg 转为 WAV。
3. 在 `voice_preview_audios.format` 写入 `wav`。
4. `audio_path` 和 `audio_url` 指向 WAV 文件。

## 用户认证接口

后端已经为 `users` 表实现注册和登录接口，前缀为：

```text
/api/auth
```

接口列表：

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| `POST` | `/api/auth/register` | 注册新账号，返回 JWT token。 |
| `POST` | `/api/auth/login` | 登录，验证用户名密码后返回 JWT token。 |
| `GET` | `/api/auth/me` | 获取当前登录用户信息（需要 `Authorization: Bearer <token>`）。 |

认证机制：

- 密码使用 bcrypt 哈希存储，不存明文。
- JWT 使用 HS256 算法，有效期 24 小时。
- 前端将 token 存储在 `localStorage` 的 `auth_token` 键中。
- 路由守卫检查 `auth_token` 是否存在，保护 `/profile` 页面。

## 个人专属音色与 owner 鉴权

`voices.owner_id` 外键引用 `users.id`：

- `owner_id IS NULL`：系统默认音色，所有用户可见，**不可编辑/删除**。
- `owner_id = <user_id>`：个人专属音色，**仅该用户可见、可编辑、可删除**。

`GET /api/voices` 匿名访问时仅返回系统默认音色；登录后返回系统默认 + 当前用户的专属音色。`POST` 创建音色时 `owner_id` 自动从 JWT token 中获取。`PUT`/`DELETE` 对 `owner_id IS NULL` 的音色返回 403。用户 A 看不到用户 B 的专属音色。

## 试听缓存写入逻辑

`POST /api/tts/preview` 当前使用数据库的流程如下：

1. 使用请求里的 `voiceId` 查询 `voice_provider_profiles.provider_voice_id`。
2. 未找到 provider 时返回 `404 voice provider not found`，避免无效音色直接进入 Edge-TTS。
3. 对试听文本计算 SHA-256，得到 `sample_text_hash`。
4. 查询 `voice_preview_audios` 中是否存在同一 provider、同一文本 hash、同一格式且 `status = ready` 的记录。
5. 如果存在，直接返回数据库中的 `audio_url` 和 `duration_seconds`。
6. 如果不存在，调用 Edge-TTS 生成 MP3。
7. 生成成功后写入 `voice_preview_audios`：
   - `voice_provider_profile_id`
   - `sample_text_hash`
   - `sample_text`
   - `format`
   - `audio_path`
   - `audio_url`
   - `duration_seconds`
   - `file_size_bytes`
   - `status = ready`
8. 返回新生成的 `audio_url`。
