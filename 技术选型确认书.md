# 有声读物智能生成系统技术选型确认书

## 选型目标

本项目优先选择适合 4 周实训周期、容易联调、运行成本低、交付风险可控的技术方案。技术栈参考《有声读物智能生成系统项目解决方案文档》，并按传统前端、传统后端、AI 工程师三类分工确认。

## 总体技术栈

| 层次 | 确认技术 | 用途 | 选型理由 |
|-----|----------|------|---------|
| 前端框架 | Vue 3.5 + TypeScript 5.7 + Vite 8 | Web 创作者平台 | 组合式 API 成熟，开发效率高，适合中小型应用快速交付 |
| UI 与样式 | Element Plus 3 + TailwindCSS 4 | 页面组件与样式构建 | Element Plus 提供稳定表单/弹窗/表格组件，TailwindCSS 便于快速定制布局 |
| 状态管理 | Pinia 3 | 用户状态、创作参数、任务状态 | Vue 官方推荐状态管理方案，轻量易维护 |
| 音频播放 | Howler.js | 试听、播放、暂停、进度控制 | 轻量稳定，适合 Web 音频播放场景 |
| 后端框架 | FastAPI 0.115+ | REST API、任务接口、业务服务 | Python 生态便于和 TTS、音频处理模块集成 |
| 数据库 | MySQL 8.0 | 用户、作品、音色、场景模板数据 | 关系模型清晰，适合业务数据管理 |
| 缓存/队列 | Redis 7.0+ / 8 | 缓存、任务状态、Celery Broker | 支撑异步生成任务和进度查询 |
| 异步任务 | Celery + Redis | 长文本 TTS、音频合成任务 | 避免生成过程阻塞 HTTP 请求 |
| TTS 服务 | Edge-TTS 首选，阿里云基础 TTS 备选 | 文本转语音 | Edge-TTS 免费、无需 GPU；阿里云 TTS 用于稳定性备选 |
| 音频处理 | pydub + FFmpeg | 拼接、BGM、音量、格式转换 | 能覆盖基础音频后处理需求 |
| 部署 | Docker 27 | 服务容器化部署 | 环境一致，便于演示和迁移 |

## 技术选型确认：按角色落地

| 角色 | 必选技术 | 可选技术 | 不建议优先投入 |
|-----|----------|----------|----------------|
| 传统前端工程师 | Vue 3.5、TypeScript、Vite、Element Plus、Pinia、Axios、Howler.js | TailwindCSS、Day.js | 复杂可视化框架、重型低代码平台 |
| 传统后端工程师 | FastAPI、Pydantic、SQLAlchemy、MySQL、Redis、Celery、JWT | Docker、WebSocket、对象存储 OSS | 过早引入微服务、复杂权限系统 |
| AI 工程师 | Edge-TTS、pydub、FFmpeg、文本规则处理、Celery Worker | 阿里云基础 TTS、BGM 素材库、敏感词库 | 自训练大模型、本地 GPU TTS 训练 |

## 关键决策

1. 后端主方案确认使用 FastAPI，而不是 Spring Boot。原因是 AI 与音频处理主要基于 Python 生态，减少跨语言集成成本。
2. TTS 首选 Edge-TTS。原因是无需 GPU、无需密钥、成本低，适合实训阶段快速演示。
3. 任务生成必须异步化。长文本合成和音频拼接耗时较长，应通过 Celery + Redis 处理，前端通过任务状态接口展示进度。
4. 音频导出优先支持 MP3，WAV 作为可选格式。MP3 文件体积更小，更适合普通用户下载和分享。
5. 数据库优先使用 MySQL 8.0。SQLite 可用于早期原型，但正式演示建议使用 MySQL + Redis。

## 风险与备选方案

| 风险 | 影响 | 备选方案 |
|-----|------|---------|
| Edge-TTS 调用不稳定 | 生成失败或超时 | 增加失败重试，接入阿里云基础 TTS 作为备选 |
| 长文本合成耗时较长 | 前端等待时间过长 | 文本分段、异步任务、进度查询、后台生成 |
| BGM 与人声音量冲突 | 成品听感差 | pydub 自动降低 BGM 音量，并提供前端音量调节 |
| 文件体积过大 | 下载慢、存储压力高 | 默认导出 MP3，WAV 作为高级选项 |
| 前后端接口频繁变更 | 联调成本上升 | 尽早确定 OpenAPI/接口文档和统一响应格式 |

## 确认结论

本项目最终确认采用 Vue 3 + FastAPI + Edge-TTS + pydub/FFmpeg + MySQL + Redis + Celery 的技术路线。该方案能覆盖有声读物智能生成系统的核心流程，并在开发效率、运行成本和演示稳定性之间取得平衡。
